# CLAUDE.md

Guidance for Claude Code and automated agents working in this repository.

## Project Overview

VS Code extension for the Verse programming language (UEFN / Unreal Editor for Fortnite). It detects missing imports from Verse compiler diagnostics, auto-inserts high-confidence `using` statements, offers multi-option quick fixes, and manages import organization in `.verse` files.

## Commands

```bash
npm run compile          # Compile TypeScript
npm run watch            # Watch mode for development
npm test                 # Run jest test suite
npx jest path/to.test.ts # Run a single test file
npm run parse-digest     # Regenerate precompiled digest data
vsce package             # Create .vsix package
```

There is no lint script. Manual smoke testing is done via F5 (Extension Development Host).

## Architecture

### Entry Point

`src/extension.ts` activates on `onLanguage:verse`. It wires together handlers, providers, and services, registers commands, and sets up file watchers and event listeners.

### Handler/Provider/Service Pattern

- **Handlers** react to VS Code events: `DiagnosticsHandler` (processes compiler errors), `CommandsHandler` (all user commands)
- **Providers** implement VS Code APIs: `ImportCodeActionProvider` (quick fixes), `ImportCodeLensProvider` (path conversion UI)
- **Services** manage data/caching: `DigestParser` (API digests), `AssetsDigestParser` (project asset classes), `ProjectPathCache`/`ProjectPathTreeBuilder` (module resolution), `PrecompiledDigestLoader`

### ImportHandler Facade

`ImportHandler` coordinates three specialized classes. Always use the facade; don't call internals directly:

- `ImportSuggestionExtractor` - parses Verse compiler errors into import suggestions. Pattern matching order matters: multi-option patterns first, then single-option, then digest lookup
- `ImportFormatter` - syntax formatting (curly vs dot), grouping (digest vs local), `isModuleImport()` detection
- `ImportDocumentEditor` - document modifications, import block management, spacing

### Auto-Import Flow

1. `onDidChangeDiagnostics` fires in `extension.ts`
2. `DiagnosticsHandler.handle()` debounces (default 3000ms) and prevents parallel processing per document
3. `ImportSuggestionExtractor.extractImportSuggestions()` parses each diagnostic message
4. Single high-confidence suggestions are auto-imported; multi-option scenarios follow the `multiOptionStrategy` setting
5. `ImportCodeActionProvider` independently provides quick fix CodeActions

### Key Data Types

`ImportSuggestion` is the core type flowing through the system:

- `source`: `'error_message'` | `'digest_lookup'` | `'inference'`
- `confidence`: `'high'` | `'medium'` | `'low'` (only `high` is auto-imported)

### Verse Language Details

Three equivalent `using` styles (all standard Verse): braced `using { /Verse.org/Simulation }`, dotted `using. /Verse.org/Simulation`, and indented `using:` + content on the next line.

`using` has two meanings: module imports (paths starting with `/` or containing `.`) vs local-scope using (bare identifiers). `ImportFormatter.isModuleImport()` distinguishes them by content; the indented style requires the next line for detection.

### External Files and Project Detection

- `Assets.digest.verse` lives outside the workspace under `%LOCALAPPDATA%\UnrealEditorFortnite\Saved\VerseProject\{ProjectName}\...` (with a legacy fallback path)
- `ProjectPathHandler` finds `.uefnproject` by searching the workspace and up to 5 parent directories; supports both new (`<project>/Content/...`) and old (`<project>/Plugins/<ProjectName>/Content/...`) structures

## Configuration

All settings live under `verseAutoImports.*` in `package.json` contributes: `general.*` (auto-import toggle, debounce), `behavior.*` (syntax style, grouping, sorting, multi-option strategy), `quickFix.*`, `pathConversion.*`, `experimental.*` (digest lookup), `cache.*`.

Legacy migration (v0.6.0): `verseAutoImports.autoImport` moved to `verseAutoImports.general.autoImport`; `general.diagnosticDelay` is deprecated in favor of `general.autoImportDebounceDelay`.

## Code Style

- TypeScript, compiled with `tsc` (no bundler). No path aliases; use relative imports within `src/`
- **No emojis** in code, comments, docs, commit messages, or release notes
- **Never add AI attribution** (no Co-Authored-By trailers, no "generated by" notes) in commits, code, or docs
- Follow [Keep a Changelog](https://keepachangelog.com/) in CHANGELOG.md; every user-facing change adds an entry under `[Unreleased]`
- Commit prefixes: `feat:`, `fix:`, `refactor:`, `chore:`, `docs:`, `test:`, `ci:`, `deps:`

## Testing

- Jest with ts-jest; the `vscode` module is mocked via `src/__mocks__/vscode.ts` (wired in `jest.config.ts`)
- Tests are co-located in `__tests__/` directories next to the code under test
- Bug fixes should include a regression test when the affected code is testable without the VS Code runtime (extractors, formatters, parsers are all testable)

## Git Workflow

- **main**: production releases (tagged `vX.Y.Z`)
- **develop**: integration branch; all feature/fix PRs target `develop`
- Branch prefixes: `feature/`, `fix/`, `refactor/`, `hotfix/` (hotfixes branch from `main`)
- Releases: a release PR from `develop` (or `release/vX.Y.Z`) into `main` bumps the version and finalizes the changelog; publishing the GitHub Release triggers Marketplace publish automatically

## Rules for Automated Agents

These apply to any Claude Code run triggered from GitHub Actions (issue triage, PR review, maintenance, @claude mentions):

- Never push directly to `main` or `develop`; always work on a branch and open a PR
- Never bump the `version` field in package.json outside release preparation
- Never close issues; label and comment instead, and leave closing to the maintainer
- Never edit files under `src/utils/*.digest.verse` by hand; they are generated by `npm run parse-digest`
- Before opening a PR, run `npm run compile` and `npm test` and report the results in the PR description
- Keep comments on issues/PRs short and factual; one comment per run unless replying to a human
